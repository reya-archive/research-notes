Open WebUI + Amazon Bedrock

오픈소스 채팅 화면(Open WebUI) 을 사내 서버에 직접 띄우고, AWS Bedrock 의 Claude AI 모델을 호출해 답변을 만드는 조합입니다. 데이터는 모두 사내에 머무르고, 토큰 사용량만 AWS 에 과금됩니다.

영업팀 핵심 3줄 ① 업로드 문서 · 대화 기록은 모두 사내 서버에 보관됩니다 - AWS 로 전송되는 건 AI 호출 본문뿐. ② 1인당 월 비용은 약 $1 ~ $3 (AWS 토큰 단가 기준, 인프라 비용 별도)으로 가장 저렴합니다. ③ AI 모델은 Claude · Nova 등 자유롭게 선택 · 교체 가능합니다 (벤더 락인 없음).

제품 개요

구성Open WebUI (사용자 채팅 화면) + LiteLLM (AI 호출 중계) + Apache Tika (문서 변환) + Amazon Bedrock (AI 모델)
제공사오픈소스 커뮤니티 + AWS
카테고리OSS 셀프호스팅 채팅 UI + AWS 매니지드 AI 모델 조합
라이선스3개 오픈소스 모두 사내 사용 자유 (라이선스 비용 0원)
핵심 가치사내 보관 + 벤더 락인 없음 + 답변 품질 세밀 조정 가능

아키텍처

사용자 → 채팅 화면 → AI 호출 중계 → AWS AI 모델 4단계로 동작합니다. 업로드 문서는 자동으로 텍스트로 변환되어 사내 서버에 저장되며, AI 호출 시 관련 부분만 추려 AWS 로 전송됩니다.

  • LiteLLM - Bedrock 의 네이티브 API 를 OpenAI 호환으로 래핑. 임베딩 · 리랭크 · 채팅 모두 같은 엔드포인트에서 처리
  • Apache Tika - DOC/XLSX/PPTX/PDF/MD 등 업로드 파일의 텍스트/표 추출. latest-full 이미지는 Tesseract OCR 포함
  • Amazon Bedrock - LLM/임베딩/리랭크 모델 세트. 토큰 · 건당 과금, 데이터 학습 미사용

구성 요소

4가지 구성 요소가 각각 다음 역할을 담당합니다.

구성 요소역할
Open WebUI 사용자가 보는 채팅 화면. 문서 업로드 · 답변 받기 · 답변 품질 조정용 관리자 화면 제공
LiteLLM 여러 AI 모델(Claude · GPT · Nova 등) 을 하나의 방식으로 호출하기 위한 중계 서버. 모델 교체가 자유로움
Apache Tika 업로드한 워드/엑셀/파워포인트/PDF 를 자동으로 텍스트로 변환. 스캔 PDF 도 글자 인식 가능
AWS Bedrock 실제 AI 답변을 만드는 AWS 모델 서비스. 사용한 만큼만 과금되며 학습에 사용되지 않음
기술팀 참고 - Open WebUI 운영 주의사항
  • Knowledge 단위로 문서를 모아 RAG 질의에 첨부 (# 또는 + 버튼)
  • RAG 튜닝 옵션 22개 (청크 크기 · Hybrid Search · 리랭커 · Top K 등) 를 Admin Panel 에서 UI 로 제어
  • PersistentConfig 함정 - 상당수 설정이 첫 부팅 시 env 로 seed 된 뒤 DB 값이 우선이 됨 → 변경은 UI 에서 해야 반영됨. 영업팀 영향: 운영 중 설정 변경은 관리자 화면에서만 가능
  • JWT 서명용 WEBUI_SECRET_KEY 는 반드시 고정값 (미설정 시 재시작마다 세션 만료)
  • 가입 최초 1인이 자동 관리자, 이후는 pending 으로 들어와 관리자 승인 필요 (ENABLE_SIGNUP=true, DEFAULT_USER_ROLE=pending)
기술팀 참고 - LiteLLM Proxy 설정
  • /v1/chat/completions · /v1/embeddings · /rerank 세 엔드포인트를 단일 키 (LITELLM_MASTER_KEY) 로 제공
  • OpenAI 전용 파라미터 (logit_bias 등) 는 drop_params: true 로 자동 제거
  • num_retries: 10, request_timeout: 600 으로 Bedrock 429 스로틀링을 지수 백오프로 흡수
  • 모델 블록에 rpm/tpm 하드 캡 추가 가능 (본 번들 기본값은 재시도 강화 방식)
기술팀 참고 - Apache Tika 설정
  • latest-full 이미지 - Tesseract · 언어 팩 포함 → 스캔 PDF OCR 가능
  • JVM heap 기본 -Xmx1g. 대형 XLSX/PDF 를 많이 올릴 때는 -Xmx2g ~ 4g 로 상향
  • 헬스 체크 - http://tika:9998/tika (컨테이너 내부 DNS)
기술팀 참고 - Bedrock 호출 모델 정식명
  • Claude Sonnet 4.6 (bedrock/global.anthropic.claude-sonnet-4-6) - 채팅 합성. Inference Profile 필수
  • Claude Haiku 4.5 - 쿼터 회피 · 비용 절감 대안. 한국어 톤 유사
  • Titan Embeddings v2 (amazon.titan-embed-text-v2:0) - 청크 임베딩. Inference Profile 불필요
  • Cohere Rerank v3.5 (cohere.rerank-v3-5:0, full ARN 호출) - Hybrid 리랭커. IAM 에 bedrock:Rerank 별도 필요

RAG 동작 원리

업로드한 문서를 작은 단위로 나눠 저장해 두고, 사용자가 질문하면 가장 관련성 높은 부분만 골라 AI 에 전달해 답변을 받습니다. 답변에는 어떤 파일에서 가져왔는지 [출처] 가 함께 표시됩니다.

기술팀 참고 - 6단계 파이프라인 상세
  1. 업로드 - 드래그&드롭 → 파일 → Tika 파싱 → 텍스트/표 추출
  2. 청킹 - token 기준 · 800 토큰 · overlap 150 · Markdown Header Splitter ON · Chunk Min Size Target 400
  3. 임베딩 - Titan Embed v2 (LiteLLM 경유) → ChromaDB (Open WebUI 볼륨 내장)
  4. 질의 - 사용자 질문 → Titan 임베딩 → Hybrid (Vector + BM25, 가중치 0.5) Top K 8
  5. 리랭크 - Cohere Rerank v3.5 → Top K 리랭커 4
  6. 합성 - Claude Sonnet 4.6 + 한국어 커스텀 RAG 템플릿 → [출처: ...] 인용 포함 답변

서버 사양

초기 시연 단계는 AWS EC2 t3.large (월 약 8만원) 로 시작하면 충분합니다. 동시 사용자 50명 이상으로 늘어나면 m6i.xlarge 또는 그 이상 으로 단계별 확장 가능합니다. 서버는 한 대로 시작해 사용량 증가에 따라 키우는 방식이 가장 비용 효율적입니다.

기술팀 참고 - 단계별 EC2 사양 권장표

현재 openwebui-hosting 번들(Docker Compose 로 Open WebUI + LiteLLM + Tika 세 컨테이너를 한 호스트에 올리는 구성) 을 AWS EC2 에 얹을 때의 단계별 권장 사양입니다. Bedrock 은 동일 리전(ap-northeast-1) 에서 호출하는 것을 전제로 합니다.

단계 인스턴스 vCPU / RAM EBS (gp3) 동시 사용자 비고
PoC 최소 t3.medium 2 / 4 GB 30 GB 1 ~ 3 엑셀 대량 업로드 시 swap 발생 가능. Tika heap -Xmx512m 제한 권장. 단기 시연 용도만
시연 권장 t3.large 2 / 8 GB 50 GB 5 ~ 10 본 번들 기본 권장. Tika heap 1g 여유. Titan 429 는 Admin Panel 안정 모드 + LiteLLM num_retries 로 흡수
소규모 운영 t3.xlarge 4 / 16 GB 100 GB 20 ~ 30 대형 PPTX / 200p+ PDF 파싱 안정. 버스트 크레딧 소진 걱정 줄어듦
중대형 운영 m6i.xlarge 4 / 16 GB 100 GB 30 ~ 50 AWS Nitro, 예측 가능 성능. 실제 운영 단계에서 권장
대용량 KB m6i.2xlarge 8 / 32 GB 200 GB 50+ 엑셀 · PPTX 배치 업로드, ChromaDB 수십만 청크 · Tika OCR 모델 캐시 포함

EBS · 스토리지 산정 근거

  • Open WebUI 볼륨 (openwebui-hosting-data) - 대화 기록 + 사용자 DB + ChromaDB(벡터) + 업로드 원본. Knowledge 1 GB 원본 기준 ChromaDB 는 약 1.5 ~ 3 GB
  • Tika 이미지 캐시 - latest-full 이미지만 2 ~ 3 GB (Tesseract · 언어 팩)
  • Docker 레이어 - 총 8 ~ 12 GB 여유 필요
  • 사내 SI 프로젝트 한 건 분량(문서 수백 건 · 수 GB) 이면 gp3 50 ~ 100 GB 로 시작, 증가 시 온라인 확장

리전 · 네트워크 · 보안 그룹

  • 리전 - ap-northeast-1 (Tokyo). Bedrock 과 동일 리전 고정. cross-region 트래픽 비용/지연 방지, 데이터 도쿄 상주 유지
  • 보안 그룹
    • inbound 13000 (Open WebUI) / 14000 (LiteLLM) 을 사내 VPN · Office IP 대역으로 제한
    • 0.0.0.0/0 전체 오픈 금지 (Admin Panel 로 바로 접근 가능한 구조라 민감)
    • outbound 443 → bedrock-runtime.ap-northeast-1.amazonaws.com 만 열어도 동작. VPC Endpoint 사용 시 더 안전
  • IAM - Bedrock InvokeModel · InvokeModelWithResponseStream · Rerank 3개 액션 필요. Resource 에 inference-profile/* 포함 (Claude 호출 전제)
  • 도메인 - 사내 DNS + 내부 CA 인증서로 HTTPS 종료 (nginx · ALB · Caddy 중 택)

확장 경로

  • 로컬 임베딩 전환 - Bedrock Titan 쿼터 부담이 크거나 완전 오프라인 시 → g5.xlarge (A10G 24GB) 에 BGE-M3 로컬 임베딩, LiteLLM 에서 Ollama 모델로 라우팅
  • 컨테이너 분리 - Fargate 또는 ECS 로 분리해 Tika 는 CPU 집약, Open WebUI 는 메모리 집약 리소스 분배
  • GPU 합성 - Claude 대신 로컬 Llama 계열 호스팅이 필요하면 g5.2xlarge 이상, 단 Bedrock 대비 운영 부담 급증
  • 멀티 AZ - EBS 볼륨 → RDS (PostgreSQL + pgvector) 로 빼고 Open WebUI 를 stateless 로 전환 후 ALB 뒤 2+ 노드

단일 노드 권장 MCNC 의 초기 SI 프로토타입 단계에서는 분산 구성보다 단일 EC2 (t3.large → m6i.xlarge) 로 운영하면서 백업만 정기적으로 받는 구성이 가장 비용 효율적입니다. 동시 50명 이상으로 성장할 때 확장 경로를 검토하세요.

데이터 처리 · 보안

  • 사내 보관 - 업로드 파일 · 검색 인덱스 · 대화 기록은 모두 사내 서버 안에 머무릅니다. AWS 로 전송되는 것은 AI 호출 시점의 본문(질문 + 검색된 문서 일부)뿐
  • 데이터 저장 위치 선택 - AI 호출 시 데이터가 어디까지 흘러가는지 3단계로 선택할 수 있습니다.
    • 도쿄 우선 - 일본 도쿄 리전에서만 처리 (데이터 상주 최우선, 처리 용량 가장 제한적)
    • 아시아 태평양 - 도쿄 + 시드니 등 아시아권에서 처리 (중간 타협점)
    • 전세계 - 미국 포함 전체 리전에서 처리 (처리 용량 가장 여유, 시연 기본값)
  • 네트워크 격리 - VPC 내부 전용 통로(VPC Endpoint) 로 인터넷을 거치지 않고 AWS 호출 가능
  • 데이터 학습 미사용 - AWS Bedrock 약관상 고객 데이터를 모델 학습에 사용하지 않음, 입출력 저장 없음
  • 국제 보안 인증 - Bedrock ISO 27001 · SOC 1/2/3 · HIPAA BAA 보유
기술팀 참고 - 보안 그룹 · IAM · Inference Profile prefix 상세
  • Bedrock Inference Profile prefix
    • jp. - 도쿄 리전 내 고정 (데이터 상주 최우선, 쿼터 가장 타이트)
    • apac. - 아시아-태평양 크로스 리전 (중간 타협점)
    • global. - 전 세계 크로스 리전 (쿼터 여유, 데이터가 미국까지 흐를 수 있음). 본 번들 시연 기본값
  • 운영 전환 시 - 고객사 요구에 따라 global. → jp./apac. 로 일괄 교체. openwebui-hosting/docs/01-prerequisites.md 참고
  • VPC Endpoint - Bedrock · STS 두 개 Endpoint 생성 시 인터넷 게이트웨이 없이도 호출 가능
  • PersistentConfig 함정 - Admin Panel 에서 설정을 바꾼 뒤에는 docker-compose.yml 의 env 를 고쳐도 반영 안 됨. 운영 매뉴얼에 명시 필수

강점과 약점

강점 5

  • 모든 문서 · 대화 기록이 사내 서버 안에만 머무름 (외부 전송 없음)
  • 저비용 - 50명 × 6개월 기준 서버 포함 약 200만원대, AI 사용량은 사용한 만큼만
  • AI 모델 자유 선택 - Claude · Nova · 국산 LLM 등으로 언제든 교체 가능
  • 답변 품질을 관리자 화면에서 세밀하게 조정 가능 (검색 방식 · 답변 깊이 등 22개 옵션)
  • MCNC 기존 자산 (mcnc-rag · Docker Compose · Bedrock 인프라) 그대로 재사용 가능

약점 5

  • 운영 중 설정 변경은 관리자 화면에서만 가능 (직접 파일 수정은 무효, 운영 매뉴얼 필수)
  • 복잡 레이아웃 PDF · 표는 Microsoft · Google 의 자동 인식 대비 다소 아쉬움
  • 대량 문서를 한꺼번에 올릴 때 AWS 처리 속도 제한 발생 가능 (안정 모드 설정으로 회피)
